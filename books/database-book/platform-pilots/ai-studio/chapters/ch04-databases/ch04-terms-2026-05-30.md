| Term / Concept | Definition | Business Significance | Examples |
|---|---|---|---|
| **CHECK Constraint** | A rule that restricts a field to a defined range or condition before a value can be stored. | Prevents invalid values from entering operational records and later distorting reports or decisions. | `CHECK (Score BETWEEN 0 AND 100)` in the chapter's `STUDENT_GRADE` example. |
| **Concurrency** | The ability of a DBMS to manage two or more users or applications reading or changing data at the same time. | Protects shared data when multiple people or systems work with the same records. | A server database managing two staff members updating appointment records. |
| **Constraint** | A DBMS-enforced rule that protects validity, identity, or relationships in stored data. | Makes data quality structural rather than dependent only on user memory or careful typing. | `NOT NULL`, `UNIQUE`, `PRIMARY KEY`, `FOREIGN KEY`, and `CHECK`. |
| **Data Dictionary** | A readable collection of metadata that documents tables, columns, data types, keys, and rules. | Helps people understand and govern the structure of a database before using it for reports or decisions. | A table listing `STUDENT_GRADE.StudentID` as a foreign key to `STUDENT`. |
| **Data Independence** | The ability to change physical storage details without changing the user's logical view of the data. | Lets a DBMS improve storage or speed while keeping applications, forms, and queries stable. | Adding an index so grade lookups run faster without changing the grade-entry form. |
| **Data Integrity** | The condition in which stored data remains accurate, consistent, valid, and trustworthy. | Supports reliable reports, dashboards, KPIs, and managerial decisions. | Rejecting an impossible score such as `150` before it enters the Grading Database. |
| **Data Processing** | The work of retrieving, filtering, summarizing, updating, or analyzing stored data. | Turns stored records into usable information for operations and decisions. | Calculating an average score by deliverable type. |
| **Data Storage** | The organized recording and preservation of data so it can be retrieved and reused later. | Gives organizations a stable record of activity instead of scattered temporary files. | Storing student, deliverable, and score records in database tables. |
| **Database** | A structured collection of related data designed for reliable storage, retrieval, and management. | Serves as an official record of activity that can support queries, reports, applications, and decisions. | The Grading Database tables used throughout the chapter. |
| **Database Application** | The interface layer through which users work with database data. | Lets users enter, view, and report on data without directly editing raw storage structures. | Access forms, dashboards, query tools, and reports. |
| **Database Approach** | A way of organizing data in shared, structured tables with metadata, keys, constraints, and DBMS control. | Replaces scattered files and fragile spreadsheets with governed data that can be reused consistently. | Separating student facts from grade facts instead of repeating everything in one sheet. |
| **Database Management System** | The software engine that creates, manages, queries, secures, and administers databases. | Provides the services that individual files lack, including integrity enforcement, querying, shared access, backup, and recovery. | Microsoft Access, SQLite, PostgreSQL, and Supabase. |
| **Database System** | The full arrangement of users, applications, the DBMS, and the database itself. | Shows that useful database work depends on people, interfaces, software, and stored structure working together. | A professor using a grade-entry form that sends data through Access into database tables. |
| **File Environment** | An older data-management approach in which separate files are maintained independently by departments, programs, or users. | Creates redundancy, inconsistency, and fragile reporting when data must be shared across an organization. | Sales and billing files storing different versions of a customer's address. |
| **Foreign Key** | A field in one table that references the primary key of another table. | Preserves relationships across tables and helps prevent orphaned or mismatched records. | `STUDENT_GRADE.StudentID` pointing to `STUDENT.StudentID`. |
| **Index** | A storage structure that helps the DBMS find rows more efficiently. | Improves retrieval speed while preserving the same logical table view for users. | An index added to speed up grade lookups. |
| **Logical View** | The way data appears to users and applications through tables, columns, relationships, and query results. | Lets users work with meaningful names and structures without managing storage details. | Seeing `STUDENT` and `STUDENT_GRADE` tables in Access. |
| **Metadata** | Data about the database structure, such as table names, column names, data types, keys, and constraints. | Makes the database self-describing and easier to govern, maintain, and explain. | A data dictionary row showing `Score` as numeric with a 0-100 rule. |
| **NOT NULL Constraint** | A rule that requires a field to contain a value. | Prevents important business facts from being left blank when the record cannot be useful without them. | Requiring `StudentID` or `Score` in a grade record. |
| **Physical View** | The behind-the-scenes way data is stored, indexed, and optimized by the DBMS. | Allows administrators and the DBMS to improve storage without changing user-facing forms or queries. | Moving database files to faster storage or adding indexes. |
| **Primary Key** | The chosen field or set of fields that uniquely identifies each row in a table. | Gives every record a stable identity so it can be found, updated, and referenced reliably. | `STUDENT.StudentID` identifying one student row. |
| **Program-Data Dependence** | A file-environment problem in which programs are tightly tied to the structure of the files they read. | Makes systems fragile because changing a file layout can break reports or applications. | A report failing after a department changes the order of fields in a file. |
| **Query-Level Join** | A match between two tables made inside a query rather than through an enforced database relationship. | Helps students see how data can be combined while recognizing that query-time matching is not the same as referential integrity. | `qryGradebookWithWeights` matching `GRADEBOOK.DeliverableType` to `GRADE_WEIGHT.DeliverableType`. |
| **Relational Table** | A database table that stores one subject in rows and columns with clear meaning and reliable row identity. | Helps reduce duplication and prepares data for keys, constraints, joins, and later normalization. | A `STUDENT` table with `StudentID`, `FirstName`, `LastName`, `Email`, and `Birthday`. |
| **SQL (Structured Query Language)** | The standard declarative language used to work with relational databases. | Lets users and applications ask precise questions of structured data across platforms. | A `SELECT` query that retrieves students who scored above 90. |
| **UNIQUE Constraint** | A rule that prevents duplicate values in a field or combination of fields. | Protects identifiers and other values that should appear only once. | Requiring each student email to be unique. |
| **Validation Rule (Access)** | An Access table-design rule that limits allowed input for a field. | Gives Access users a visual way to apply constraint-like protection without writing SQL. | `Between 0 And 100` on the `Score` field in the Let's Build activity. |

## Acronyms and Abbreviations

| Abbreviation | Full Form | Brief Meaning | Where It Appears |
|---|---|---|---|
| **DBMS** | Database Management System | Software that creates, manages, queries, secures, and administers databases. | Database architecture and platform discussion |
| **FK** | Foreign Key | A field that connects one table to a related row in another table. | Mermaid ERD and relationship examples |
| **PK** | Primary Key | A field or fields that uniquely identify each row. | Table design, ERD notation, and key discussion |
| **SQL** | Structured Query Language | A standard language for querying and managing relational databases. | SQL preview and platform comparison |
