| Term / Concept | Definition | Business Significance | Examples |
|---|---|---|---|
| **Candidate Key** | Any field or combination of fields that could uniquely identify each row in a table. | Helps designers evaluate which columns can serve as reliable identifiers before choosing a primary key. | `StudentID` and `Email` are both candidate keys in the `STUDENT` table, but only one becomes the primary key. |
| **CHECK Constraint** | A DBMS-enforced rule that restricts a field to a defined range or condition. | Prevents impossible values from entering operational records and later distorting reports or decisions. | `CHECK (Score BETWEEN 0 AND 100)` in the chapter's `STUDENT_GRADE` example. |
| **Constraint** | A DBMS-enforced rule that protects validity, identity, or relationships in stored data. | Makes data quality structural rather than dependent only on user memory or careful typing — the system enforces the rule every time. | `NOT NULL`, `UNIQUE`, `PRIMARY KEY`, `FOREIGN KEY`, and `CHECK`. |
| **Data Dictionary** | A readable collection of metadata documenting tables, columns, data types, keys, and rules. | Helps people understand and govern the structure of a database before using it for reports or decisions. | A table listing `STUDENT_GRADE.StudentID` as a foreign key referencing `STUDENT`. |
| **Data Independence** | The ability to change physical storage details — indexing, file placement, optimization — without changing the logical view users and applications see. | Lets technical staff improve performance and storage while keeping forms, queries, and reports stable. | Adding an index so grade lookups run faster without changing the grade-entry form. |
| **Database** | A structured collection of related data designed for reliable storage, retrieval, and management. | Serves as the official record of organizational activity — one shared source of truth that supports queries, reports, applications, and decisions. | The Grading Database tables used throughout the chapter. |
| **Database Approach** | A way of organizing data in shared, structured tables with metadata, keys, constraints, and DBMS control — replacing scattered files and fragile spreadsheets. | Moves organizations from competing departmental copies to one governed, reusable data foundation. | Separating student facts from grade facts in related tables instead of repeating everything in one flat sheet. |
| **Database Management System (DBMS)** | The software engine that creates, manages, queries, secures, and administers databases. | Provides the services that individual files lack: integrity enforcement, query processing, shared access, backup, and recovery. | Microsoft Access, SQLite, PostgreSQL, and Supabase. |
| **Database System** | The full arrangement of users, applications, the DBMS, and the database itself. | Shows that useful database work depends on people, interfaces, software, and stored structure working together — not the database alone. | A professor using a grade-entry form that sends data through Access into database tables. |
| **Deletion Anomaly** | A flat-table problem where removing one row unintentionally removes an unrelated fact, because multiple themes are mixed together. | Can erase the only record that a customer, student, or product ever existed, creating permanent gaps in organizational memory. | Deleting the only grade for a student also removes the only record that the student was ever enrolled. |
| **File Environment** | An older data-management approach in which separate files are maintained independently by departments, programs, or users. | Creates redundancy, inconsistency, program-data dependence, and governance conflicts when data must be shared across an organization. | Sales and billing departments storing different versions of the same customer's address in separate spreadsheets. |
| **Foreign Key** | A field in one table that references the primary key of another table. | Preserves relationships across separated tables and prevents orphaned records — every reference must point to something real. | `STUDENT_GRADE.StudentID` referencing `STUDENT.StudentID`. |
| **Index** | A storage structure that helps the DBMS find rows more efficiently without changing the logical table view. | Improves query speed while preserving data independence — users and applications see the same tables regardless of indexing changes. | An index added to `StudentID` so grade lookups by student run faster. |
| **Insertion Anomaly** | A flat-table problem where you cannot add one fact without also adding an unrelated fact, because multiple themes are forced into one structure. | Blocks routine business operations — you cannot record a new customer, product, or student until a related transaction exists to carry it. | You cannot record a new student in a flat grade file until that student receives a grade. |
| **Logical View** | The way data appears to users and applications — tables, columns, relationships, and query results. | Lets users work with meaningful names and structures without needing to manage or understand physical storage. | Seeing `STUDENT` and `STUDENT_GRADE` tables in Access. |
| **Metadata** | Data about the database structure: table names, column names, data types, keys, and constraints. | Makes the database self-describing — people and systems can discover what data exists and what rules apply before using it. | A data dictionary row showing `Score` as numeric with a `CHECK (Score BETWEEN 0 AND 100)` rule. |
| **Natural Key** | A real-world attribute — such as email, SSN, or product code — used to identify records. | Can work when the value is guaranteed unique and stable, but natural keys often change, which is why surrogate keys are usually preferred. | Using `Email` as a primary key in a `STUDENT` table — works until a student changes their email. |
| **NOT NULL Constraint** | A rule that requires a field to contain a value — the field cannot be left empty. | Prevents important business facts from being missing when the record cannot be useful without them. | Requiring `StudentID` or `Score` in a grade record. |
| **Physical View** | The behind-the-scenes way data is stored, indexed, and optimized by the DBMS — hidden from users. | Allows administrators to improve storage, add indexes, or reorganize files without disrupting user-facing forms or queries. | Moving database files to faster storage or adding indexes behind the scenes. |
| **Primary Key** | The chosen field or set of fields that uniquely identifies each row in a table and cannot be NULL. | Gives every record a stable, unique identity so it can be found, updated, and referenced reliably — the foundation of all table relationships. | `STUDENT.StudentID` identifying exactly one student row. |
| **Program-Data Dependence** | A file-environment problem in which programs are tightly tied to the structure of the files they read. | Makes systems fragile — a small change to a file's layout can break every program and report that depends on it. | A grade report failing after a department changes the column order in a spreadsheet. |
| **Query-Level Join** | A match between two tables created inside a query rather than through a formal database-enforced relationship. | Helps students combine data across tables while recognizing that query-time matching is not the same as referential integrity — a foreign key constraint is stronger. | `qryGradebookWithWeights` matching `GRADEBOOK.DeliverableType` to `GRADE_WEIGHT.DeliverableType`. |
| **Relational Table** | A database table that stores one subject in rows and columns, with each row uniquely identifiable and each cell holding one atomic value. | Reduces duplication, prevents anomalies, and prepares data for keys, constraints, joins, and later normalization. | A `STUDENT` table with `StudentID`, `FirstName`, `LastName`, `Email`, and `Birthday`. |
| **SQL (Structured Query Language)** | The standard declarative language used to work with relational databases — you describe what result you want, not how to get it. | Lets users and applications ask precise questions of structured data across platforms — Access, SQLite, PostgreSQL, and beyond. | `CREATE TABLE STUDENT (StudentID INTEGER PRIMARY KEY, Email TEXT NOT NULL UNIQUE);` |
| **Surrogate Key** | A system-generated identifier — such as an auto-number — that has no business meaning and serves only to identify rows. | Provides a stable, unchanging identifier independent of real-world attributes that may change over time. | `StudentID` as an auto-numbered primary key in the `STUDENT` table. |
| **UNIQUE Constraint** | A rule that prevents duplicate values in a field or combination of fields. | Protects identifiers, emails, and other values that should appear only once — preventing duplicate records and lookup failures. | Requiring each student email to be unique in the `STUDENT` table. |
| **Update Anomaly** | A flat-table problem where changing one fact requires editing it in many rows, and missing even one copy creates inconsistency. | Turns routine changes — a new email, a corrected name — into error-prone multi-step operations that undermine trust in the data. | Updating a student's email address in every grade row instead of once in a `STUDENT` table. |

## Acronyms and Abbreviations

| Abbreviation | Full Form | Brief Meaning | Where It Appears |
|---|---|---|---|
| **DBMS** | Database Management System | Software that creates, manages, queries, secures, and administers databases. | Database architecture and platform discussion |
| **FK** | Foreign Key | A field that connects one table to a related row in another table. | Key definitions, ERD notation, and relationship examples |
| **PK** | Primary Key | A field or combination of fields that uniquely identifies each row. | Table design, key discussion, and SQL `CREATE TABLE` examples |
