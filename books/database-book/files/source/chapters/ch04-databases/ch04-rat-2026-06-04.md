## Assessment Design Notes

This RAT checks whether students completed the Chapter 4 reading and can reason with database fundamentals — the DBMS/application/database architecture, flat-file anomalies, relational table rules, primary and foreign keys, constraints, and the bridge from spreadsheets to structured database systems. Questions are grounded in the Grading Database examples, the PetVax Lab 03 connections, the Let's Build Access activities, and chapter-specific SQL and table structures.

### Bloom Distribution

| Bloom Level | Questions | Count |
|---|---|---|
| Remember | 1–8 | 8 |
| Understand | 9–16 | 8 |
| Apply | 17–24 | 8 |
| Analyze | 25–32 | 8 |
| Evaluate | 33–40 | 8 |

### Design Criterion Coverage

| Design Criterion | Bloom Sections Used | Questions | Count |
|---|---|---|---|
| Application-based | Apply, Analyze, Evaluate | 17–24, 26, 28, 30, 32, 34–36, 38, 40 | 14 |

### AI-Resistance Strategies Used

1. **Chapter-specific reasoning** — questions reference the Grading Database tables (`STUDENT`, `STUDENT_GRADE`), PetVax Lab 03 scenarios (Coco, Angel, Rex, Sarah Perry), and the Let's Build Access `GRADEBOOK` table structure.
2. **Schema-specific context** — questions use exact column names, constraint types, and key definitions from the chapter's SQL and data dictionary examples.
3. **Scenario stems with embedded traps** — distractors drawn from adjacent concepts (surrogate vs. natural key, foreign key vs. primary key, logical vs. physical view, file environment vs. database approach).
4. **Multi-answer discrimination** — Select ALL questions require distinguishing database capabilities from spreadsheet features, or identifying all constraint types from a mix of real and invented ones.
5. **Non-obvious correct answers** — paraphrased chapter language rather than keyword-matched definitions (e.g., "query fragility" rephrased as "formulas that break when rows are added").
6. **Output prediction** — questions ask what would happen if a constraint were missing, a key were changed, or a flat file were updated incorrectly.
7. **Platform-specific details** — questions reference Access AutoNumber, SQLite INTEGER PRIMARY KEY, and the chapter's CREATE TABLE constraint syntax.

## Answer Key

### Remember Questions

**Question 1: What is a DBMS?**

**Correct Answer:** B

**Explanation:** Chapter 4 defines a DBMS as "the software engine that creates, manages, queries, secures, and administers databases." A database is the collection of data (A). Applications include forms and reports (C). A DBMS is not a website programming language (D).

| Option | Correct? | Reasoning |
| ------ | -------- | --------- |
| A | No | Describes a database, not the management software. |
| B | Yes | The DBMS is the software that manages, queries, and secures the database. |
| C | No | Describes applications, which sit above the DBMS in the layered architecture. |
| D | No | A DBMS is not a general-purpose web programming language. |

---

**Question 2: Which sequence correctly describes the layers of a database system from top to bottom?**

**Correct Answer:** B

**Explanation:** Chapter 4's layered diagram shows Users at the top, then Applications, then DBMS, then Database at the bottom. Each layer depends on the one below it.

| Option | Correct? | Reasoning |
| ------ | -------- | --------- |
| A | No | Reverses the layers — Users are at the top, Database at the bottom. |
| B | Yes | Users → Applications → DBMS → Database is the correct order. |

| D | No | Incorrectly places Users below Applications and omits the DBMS and Database layers. |

---

**Question 3: What is a primary key?**

**Correct Answer:** B

**Explanation:** Chapter 4 defines a primary key as a field or combination of fields that uniquely identifies each row and cannot be NULL. A field that can be empty (A) violates the NOT NULL requirement. A column referencing another table (C) describes a foreign key. Position in the table (D) does not determine key status.

| Option | Correct? | Reasoning |
| ------ | -------- | --------- |
| A | No | A primary key cannot be NULL. |
| B | Yes | Uniquely identifies each row and cannot be NULL — the chapter's definition. |
| C | No | Describes a foreign key, not a primary key. |
| D | No | Primary keys are defined by constraints, not column position. |
