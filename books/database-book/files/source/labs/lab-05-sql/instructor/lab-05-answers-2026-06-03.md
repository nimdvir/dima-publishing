## Consistency Checks (file vs. quiz)

Confirm the student's file produces values that match their quiz answers. If the quiz says one thing and the script another, flag it.

- `SELECT COUNT(*)` after setup → **24** (Q1).
- `SELECT COUNT(*)` after both inserts → **26** (Q8).
- `WHERE AppointmentStatus = 'No-show'` → **2 rows** (Q3).
- `WHERE VaccineDue = 'Yes'` → **11 rows** before inserts (Q4).
- `WHERE ReminderSent = 'No'` → **4 rows** (Q5).
- `WHERE OwnerEmail IS NULL` → **1 row** after inserts (Q9).
- `WHERE PaymentAmount >= 100` → **4 rows** after inserts (Q10).
- `GROUP BY ServiceType` count → Vaccination highest at **12** (Q11).
- `HAVING AVG(PaymentAmount) >= 100` → **Dental Cleaning, Emergency Visit** (Q12).
- `CASE` Low tier → **5 rows** (Q13).
- `SERVICE_RATES` max fee → **Emergency Visit, 439** (Q14).

## Notes for the AI Grader

- The two `INSERT` rows are the integrity anchor. A file missing the Maple (1025) and Biscuit (1026) inserts cannot produce the count of 26 or the single `IS NULL` row, so flag it.
- Accept either `ROUND(AVG(...), 2)` or plain `AVG(...)` in Step 12; the `HAVING` result is the same.
- Accept reasonable column aliases and ordering differences. Grade on correct rows returned, not exact formatting.
- The Step 15 `UPDATE` should be **commented out**. If the student ran it, `AppointmentStatus` for 1011 will read `Completed`; this does not change any graded quiz answer but should be noted.
- No query in this lab should contain a `JOIN`. Joining two tables is a Chapter 6 skill; a `JOIN` here is out of scope but not an error to penalize.
