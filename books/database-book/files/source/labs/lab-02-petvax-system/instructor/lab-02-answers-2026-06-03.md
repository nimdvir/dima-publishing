## Overall Purpose

This lab transfers the Chapter 2 logic from Let’s Build 2 to the PetVax project. Students use Dr. Dima’s handwritten notebook as messy operational evidence and build a one-page PetVax Performance Logic Map.

1. **Brightspace quiz (auto-graded):** 16 questions, 24 points. Every answer is exact, so Brightspace can grade it automatically.

# Logic Map File — Completion Check (16 points)

The Brightspace quiz already scores concept understanding. The file submission is **not** a second content rubric. It is checked for completion and authenticity: did the student actually do the work, using one consistent scenario?

The 16 points are awarded for a complete, consistent, plausible Logic Map. Score the file on the three checks below, not on per-row writing quality.

| Check | What to confirm | Points |
|---|---|---:|
| Completeness | All 13 rows of the Logic Map table are filled in with non-empty, on-topic answers | 8 |
| Scenario consistency | One PetVax problem (from the notebook) is carried through every row — the KPI, DIKW chain, process, information user/use, data-quality risk, R.E.A.D. stage, five-component risk, and IS explanation all refer to the same problem | 6 |
| Authenticity | Answers are anchored in a specific notebook line and are not generic AI boilerplate | 2 |
| **Total** |  | **16** |

**Partial credit:** subtract from Completeness for each missing or off-topic row. Subtract from Scenario consistency if the student switches problems partway through. Award Authenticity only when at least one answer quotes or paraphrases a specific notebook line.

# AI-Grading Guidance for the File

The AI grader is checking that the student did the work — not re-scoring the concepts. Follow this order:

1. **Find the anchor.** Identify the student’s `Notebook evidence` line and `PetVax performance problem`. If neither is anchored in a specific notebook line, flag the file as generic and cap Authenticity at 0.
2. **Check completeness.** Confirm all 13 rows are present and filled with non-empty, on-topic answers.
3. **Check consistency.** Confirm the KPI, DIKW chain, process, information user/use, data-quality risk, R.E.A.D. stage, five-component risk, and IS explanation all refer to the same problem. A file that switches scenarios partway through loses Scenario-consistency points.
4. **Spot generic submissions.** A submission that does not reference the notebook, that reads like a textbook summary, or that lists unrelated examples per row should receive low Authenticity and Consistency scores even if every row is filled.

The three reference patterns below show what complete, consistent submissions look like. Students do not need to match them word for word; any internally consistent problem from the notebook is acceptable.

## Strong Submission Pattern 1 — Reminder Calls

| Layer | Strong answer |
|---|---|
| Notebook evidence | `vacc reminder calls — did 5? lost track` |
| PetVax performance problem | Reminder calls are not tracked reliably |
| KPI | Reminder completion rate |
| Data: one raw fact | Five reminder calls may have been completed, but the count is uncertain |
| Information: summary or comparison | The clinic cannot confirm how many reminder calls were completed that day |
| Knowledge: pattern or explanation | When reminder calls are tracked informally, staff lose visibility into which owners were contacted |
| Wisdom: decision or action | Require staff to log each reminder call in a shared system before the end of the shift |
| Process that creates the data | Staff review vaccination reminder list → call owners → record call status → unresolved calls remain on the list |
| Information user and use | Clinic manager checks reminder completion report and follows up on missed calls |
| Data-quality risk | Completeness: reminder-call records may be missing or unfinished |
| R.E.A.D. stage most involved | Representation and Retrieval, because reminder calls must be recorded clearly before they can be tracked |
| Weakest five-component link | Process: staff do not have a consistent reminder logging routine |
| Why this is an information system | It combines people, process, data, and tools to make reminder follow-up reliable and actionable |

# Common Mistakes and Feedback

| Mistake | Suggested feedback |
|---|---|
| No notebook evidence | “Anchor your answer in one specific line from Dr. Dima’s notebook.” |
| KPI is not measurable | “Your KPI should be a measurable signal, such as reminder completion rate, follow-up scheduling completion rate, or reconciliation accuracy.” |
| Data is actually information | “A data row should be one raw fact. A summary such as ‘several notes are missing follow-up dates’ belongs in the information row.” |
| Information is actually knowledge | “Information summarizes what happened. Knowledge explains a pattern or likely cause.” |
| Wisdom is not an action | “Wisdom should name a decision or action PetVax could take.” |
| Process is missing | “Explain how the data is created. Which work activity produces the record?” |
| Information behavior is missing | “Name who needs the information, where they would look for it, and how they would use it.” |
| Data-quality risk is generic | “Tie the risk to accuracy, completeness, timeliness, or consistency.” |
| Five-component answer only says software | “Consider whether the problem could also involve people, processes, data, hardware, or software.” |
| Response switches scenarios | “Use one PetVax problem throughout the entire logic map.” |
| IS explanation is too narrow | “Explain how people, processes, data, and technology work together to support decisions.” |
